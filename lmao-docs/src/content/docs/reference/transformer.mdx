`@smoothbricks/lmao-transformer` is a **separate companion package** (not a subpath of `@smoothbricks/lmao`) that
provides a standard TypeScript custom transformer. It rewrites your traced code at compile time for source-line
attribution and zero-overhead tagging.

:::caution[Entirely optional]
No documented `@smoothbricks/lmao` API requires this transformer. Your code compiles and runs identically without it —
you simply forgo automatic source-line numbers and the `ctx.tag` inlining micro-optimization. It is a pre-1.0 companion
package; treat it as an advanced, opt-in build integration.
:::

```bash
bun add -d @smoothbricks/lmao-transformer
```

## `createLmaoTransformer(options?)`

Returns a `ts.TransformerFactory<ts.SourceFile>` for the compiler's `before` phase.

```ts
createLmaoTransformer(options?: LmaoTransformerOptions): ts.TransformerFactory<ts.SourceFile>
```

`LmaoTransformerOptions`:

| Option | Purpose |
|---|---|
| `typeChecker?` | A `ts.TypeChecker`. When provided, only calls on genuine LMAO context/logger types are rewritten (type-aware). Without it, transformation is purely structural (matches `ctx.log.*`, `ctx.span`, `ctx.tag`, `ctx.ok`/`ctx.err` by shape). |
| `projectRoot?` | Root for computing relative paths in injected metadata. Defaults to the git repository root, else `process.cwd()`. |

Wiring depends on your build tool — anything that accepts custom TS transformers (for example `ts-patch`, `ts-loader`,
or a bundler's TypeScript plugin). The self-contained form via `ts.transpileModule`:

```ts
import ts from 'typescript';
import { createLmaoTransformer } from '@smoothbricks/lmao-transformer';

const { outputText } = ts.transpileModule(source, {
  compilerOptions: { module: ts.ModuleKind.ESNext, target: ts.ScriptTarget.ESNext },
  transformers: { before: [createLmaoTransformer()] },
});
```

## What it rewrites

For code written against the shipped `ctx` surface:

- **Log lines** — `ctx.log.info('msg')` → `ctx.log.info('msg').line(N)`, where `N` is the call-site source line. The
  same applies to `debug`/`warn`/`error`/`trace`, and `.line(N)` is inserted before any further chained setters.
- **Result lines** — `ctx.ok(value)` → `ctx.ok(value).line(N)`; `ctx.err(error)` → `ctx.err(error).line(N)`.
- **Child spans** — `ctx.span('name', opOrFn)` is rewritten to a monomorphic internal dispatch that carries the
  call-site line number.
- **Tag inlining** — a `ctx.tag.*` chain used as a statement is inlined to direct columnar buffer writes (no method
  calls on the hot path):

```ts
// before
ctx.tag.operation('SELECT').userId(userId);
// after (conceptually)
{
  ctx._buffer.operation_values[0] = 'SELECT';
  ctx._buffer.operation_nulls[0] |= 1;
  ctx._buffer.userId_values[0] = userId;
  ctx._buffer.userId_nulls[0] |= 1;
}
```

Chains already carrying `.line(...)` are left untouched, and a `ctx.tag` chain assigned to a variable (rather than used
as a statement) is not inlined.

The package also exposes the tag-chain inliner building blocks directly — `tryTransformTagChain`, `findTagChainRoot`,
`extractSchemaFromTagType` (with `TagChainInfo`, `SchemaFieldInfo`) — for advanced integrations.

:::note[Legacy transforms]
The transformer additionally recognizes the older module-context API: it injects build metadata into `defineModule({…})`
calls and appends line numbers to `task('name', fn)` calls. The shipped context API is
[`defineOpContext` / `defineOp`](/reference/define-op-context/) — these legacy rewrites don't fire on current code.
:::

## Not yet implemented

The design also describes a destructured-context (`__ctx`) rewrite that hoists context members at call sites. That part
is **not implemented**; only the line injection and tag-chain inlining above ship today.
