# Three-step release process

This document describes the release workflow generated by `@smoothbricks/cli`. It is package-agnostic: repositories opt
in by declaring platform-family Nx targets and exact target outputs. The generator must not contain package names,
native filenames, or package-specific build commands.

The generated platform workflow has three jobs:

1. `linux-release-candidate` versions and validates the release;
2. `macos-platform` builds Apple artifacts in parallel with the Linux candidate;
3. `publish-on-linux` restores both producers, assembles the package, and publishes.

Those jobs form two execution stages: the two producers run concurrently, then the final consumer runs after both
succeed. Repositories without Apple-platform targets keep the compact single-job Ubuntu workflow and perform no artifact
copy.

## Invariants

A release is valid only when all of these remain true:

- `smoo release version` runs exactly once for the candidate being published.
- Each output manifest records the exact checked-out `HEAD` that produced its files.
- The Linux producer validates and bundles the exact local release commit.
- The parallel macOS producer builds the dispatch commit; release versioning must therefore remain metadata-only for
  platform compilation.
- Cross-job build outputs move through immutable workflow artifacts, never through Nx cache restoration.
- Every transferred file is declared by the destination checkout's resolved Nx project/target `outputs` metadata.
- Every transferred file is verified by source SHA, size, and SHA-256 before it enters the publishing worktree.
- Paths, symlinks, undeclared files, duplicate destinations, and destination collisions fail closed.
- The final Ubuntu job restores every file that enters the package tarball; it does not rebuild producer work.
- Release validation does not honor ordinary-CI `ci:skip:<target>` tags.
- Only the final publishing job receives trusted-publishing/OIDC capability.
- The generated `# Step N` comments remain aligned with GitHub's job-local step numbering because external deeplinks use
  those anchors.

## Platform target contract

Smoo discovers platform work from resolved Nx project metadata. A target opts into a platform family through its exact
target name:

| Target-name glob | Producer                  | Purpose                                                            |
| ---------------- | ------------------------- | ------------------------------------------------------------------ |
| `*-macos`        | `macos-platform`          | macOS-only output                                                  |
| `*-ios`          | `macos-platform`          | Apple/iOS output requiring macOS                                   |
| `*-linux`        | `linux-release-candidate` | supplemental Linux output excluded from ordinary aggregate `build` |

These families are intentionally excluded from the ordinary aggregate `build` dependencies. Otherwise a generic Linux
`nx build` could try to execute Apple targets, or an ordinary local build could unexpectedly build every release
platform.

Each platform target must:

1. have a real executor and command;
2. be cacheable when its command is deterministic;
3. declare precise workspace-relative `outputs` owned by that target;
4. name only target dependencies that are valid on the same runner family;
5. write platform-qualified filenames when multiple targets contribute to one package;
6. avoid a package-local no-op aggregate `build` target.

Example shape:

```json
{
  "nx": {
    "targets": {
      "native-arm64-macos": {
        "executor": "nx:run-commands",
        "cache": true,
        "inputs": ["{projectRoot}/src/**/*.rs", "{projectRoot}/Cargo.toml"],
        "outputs": ["{projectRoot}/dist/addon.darwin-arm64.node"],
        "options": {
          "command": "build-native --target aarch64-apple-darwin --output dist/addon.darwin-arm64.node"
        }
      },
      "native-arm64-linux": {
        "executor": "nx:run-commands",
        "cache": true,
        "inputs": ["{projectRoot}/src/**/*.rs", "{projectRoot}/Cargo.toml"],
        "outputs": ["{projectRoot}/dist/addon.linux-arm64-gnu.node"],
        "options": {
          "command": "build-native --target aarch64-unknown-linux-gnu --output dist/addon.linux-arm64-gnu.node"
        }
      }
    }
  }
}
```

The commands are illustrative. Smoo owns target discovery, execution, transfer, and validation; the package owns the
compiler invocation and its supported target triples.

## Managed workflow generation

`smoo monorepo init` resolves workspace projects through Nx, detects the canonical platform target families, and renders
the managed `.github/workflows/publish.yml`. Repositories declare targets and outputs; they do not add package-specific
workflow branches by hand.

Generation chooses one of two exact shapes:

- no `*-macos`/`*-ios` target matches: one Ubuntu job with no artifact upload, download, bundle, or overlay;
- one or more Apple target matches: the parallel producer graph below, plus explicit artifact manifests and assembly.

The checked-in workflow is validated byte-for-byte against the generator. Re-running managed-file initialization updates
the workflow when the resolved Nx target families change.

## Two-stage publish graph

The three jobs form this two-stage graph:

```text
linux-release-candidate ─┐
                         ├──> publish-on-linux
macos-platform ──────────┘
```

`linux-release-candidate` and `macos-platform` start independently and run in parallel. `publish-on-linux` is the only
job that combines their outputs and the only job allowed to publish.

## Step 1: Linux release candidate

The Linux producer owns release versioning and validation:

1. check out the workflow dispatch commit with full history;
2. restore the Linux/runner-architecture toolchain caches and enter devenv;
3. configure the release author and build the self-hosted CLI when required;
4. repair any previously interrupted release state;
5. run `smoo release version` and capture its `mode`, selected projects, and resulting release SHA;
6. check generated/managed monorepo files against the versioned candidate;
7. run ordinary aggregate `build` and collect its resolved dependency-closure outputs;
8. run matching supplemental `*-linux` targets and collect their declared outputs;
9. run `lint`, `test`, and `smoo monorepo validate` against the versioned candidate;
10. create a Git bundle containing the candidate commit and release tags;
11. upload the Git bundle and collected Linux/workspace output tree.

The two build/collection commands remain separate so their job steps and artifacts are independently diagnosable:

```sh
smoo github-ci nx-run-many \
  --targets "build" \
  --projects "$RELEASE_PROJECTS" \
  --collect-outputs "$RUNNER_TEMP/release-build-outputs"

smoo github-ci nx-run-many \
  --targets "*-linux" \
  --projects "$RELEASE_PROJECTS" \
  --collect-outputs "$RUNNER_TEMP/linux-platform-outputs"
```

`nx-run-many` expands globs only within the explicitly selected projects and groups exact targets by their owning
projects. Collection follows each selected target's resolved same-project dependency closure, so an aggregate `build`
transports the concrete `*-js`, `*-napi`, `*-wasm`, or other tool-output files it actually consumed. A clean glob
no-match is a successful no-op; an explicit empty project selection or target failure remains fatal.

The Linux manifest records the release candidate's `git rev-parse HEAD`. The Git bundle—not a remote checkout and not an
inferred version—is the source of truth for the final job.

## Step 2: macOS platform producer

`macos-platform` starts at the same time as the Linux producer. It:

1. checks out the workflow dispatch SHA;
2. restores only macOS/runner-architecture caches and enters devenv;
3. builds the self-hosted CLI when required;
4. resolves and runs matching `*-macos` and `*-ios` targets across workspace projects that own them;
5. collects only those targets' resolved declared outputs;
6. uploads the platform-output tree and manifest;
7. saves macOS/devenv caches in `always()`.

The platform invocation is generic:

```sh
smoo github-ci nx-run-many \
  --targets "*-macos,*-ios" \
  --collect-outputs "$RUNNER_TEMP/macos-platform-outputs"
```

The macOS manifest records the dispatch checkout's `git rev-parse HEAD`. It does not claim to have built the local
release commit being created concurrently on Linux.

Because release selection is being computed concurrently on Linux, the macOS producer may build matching targets for
packages outside the eventual release set. The final pack step ignores unrelated outputs. This bounded extra work is
safer than serializing the producers or independently reimplementing release selection.

Parallelism requires a strict versioning invariant: release versioning may change package/release metadata, but native
platform compilation must not consume generated version state or embed the release commit SHA. If that ever changes,
versioning must move to a preceding job and both producers must restore that candidate before compiling. Do not weaken
source-SHA checks to preserve the two-stage graph.

For `mode: none` and dry runs, the macOS job may perform unnecessary work because it cannot consume the parallel Linux
job's mode output. That is the deliberate cost of keeping producer builds parallel.

## Step 3: assemble and publish on Linux

`publish-on-linux` declares:

```yaml
needs: [linux-release-candidate, macos-platform]
runs-on: ubuntu-latest
```

It:

1. checks out the dispatch commit to obtain the repository and local actions;
2. downloads artifacts only from its two `needs` producers in the same workflow run;
3. restores the exact Linux release commit and tags from the Git bundle before devenv setup;
4. restores and applies Linux/workspace outputs against the Linux candidate SHA;
5. verifies and applies macOS outputs against the dispatch SHA;
6. validates each manifest against the restored checkout's resolved Nx project, target, output declaration, and path;
7. rejects checksum, size, SHA, ownership, undeclared-file, collision, traversal, and symlink failures before copying;
8. runs monorepo and packed-package validation against the fully assembled worktree;
9. publishes with npm trusted publishing/OIDC, or reports the dry-run result;
10. pushes the prepared release commit and tags according to the release transaction;
11. optionally deploys production only after a non-dry-run release succeeds.

The producer trees use different, explicit provenance values:

```sh
smoo github-ci apply-outputs \
  --source-sha "$RELEASE_SHA" \
  "$RUNNER_TEMP/publish-artifacts/publish-release-outputs-$GITHUB_RUN_ID" \
  "$RUNNER_TEMP/publish-artifacts/publish-linux-outputs-$GITHUB_RUN_ID"

smoo github-ci apply-outputs \
  --source-sha "$GITHUB_SHA" \
  "$RUNNER_TEMP/publish-artifacts/publish-macos-outputs-$GITHUB_RUN_ID"
```

### Artifact manifest verification

`--collect-outputs` writes an exact Typia-validated manifest next to the staged files. Each entry contains:

- owning Nx project;
- exact resolved target;
- declared output expression;
- workspace-relative destination path;
- producer source SHA;
- byte size;
- SHA-256 digest.

The manifest is not authority for its own permissions. `apply-outputs` resolves the destination checkout's Nx metadata
and requires the manifest's project, target, output expression, and path to match that current contract.

Application rejects malformed or extra manifest properties, stale SHAs, checksum/size mismatches, missing or undeclared
files, duplicate destinations, existing destination collisions, absolute paths, traversal, and symlinks in either the
artifact tree or destination path. Verification completes before any file is copied into the publishing worktree.

### Caches are not artifact transport

Nix/devenv, `node_modules`, and Nx cache keys include both `runner.os` and `runner.arch`. This prevents one architecture
from restoring incompatible native state from another.

Those caches accelerate setup and repeated tasks; they do not move release outputs between jobs. GitHub workflow
artifacts carry the candidate bundle and declared target outputs. The release remains correct when every cache misses.

No producer job receives `id-token: write`. Native compilers and artifact upload steps therefore cannot publish packages
even if compromised.

A native package must include all required platform-qualified files in its packed tarball and select them by
`process.platform` and `process.arch` at runtime. Package-specific tests must prove every claimed supported platform
maps to exactly one included filename and that missing/unsupported combinations fail explicitly.

## Repositories without Apple targets

If no selected workspace target matches `*-macos` or `*-ios`, the generator retains the existing single Ubuntu `publish`
job. The same logical gates execute inline:

1. version the candidate;
2. run ordinary build plus matching `*-linux`, lint, tests, and monorepo validation;
3. publish and optionally deploy.

No empty producer job or artifact round-trip is generated. A repository with no platform-family targets retains its
previous workflow shape.

## Ordinary CI skip tags

A project may skip an exact target only in ordinary `nx-smart` CI:

```json
{
  "nx": {
    "tags": ["ci:skip:test"]
  }
}
```

Rules:

- the form is `ci:skip:<exact-target>`;
- the target must exist on that resolved project;
- the policy is generic—`build`, `lint`, `test`, or another exact target can be named;
- `smoo monorepo validate` rejects malformed tags and missing targets;
- explicit release `nx-run-many` calls do not honor the tag.

This permits a temporary ordinary-CI exclusion without silently removing the same package from release gates.

## Generated workflow step anchors

GitHub inserts “Set up job” as step 1. Generated workflow comments therefore begin checkout at `# Step 2`.
Composite-action internals do not increment the surrounding job's top-level step number.

The comments are a compatibility contract, not decoration. External logs and deeplinks use the job-local step number.
Any renderer change that inserts, removes, or conditionally emits a top-level step must regenerate every following
`# Step N` comment and update exact rendering tests. A skipped conditional step still exists in the job definition and
therefore still occupies its number.

Before accepting a generated workflow change, verify:

- every top-level step has the correct preceding `# Step N` comment;
- numbering restarts for each job;
- the automatic GitHub step remains documented as step 1;
- single-job and platform workflows both preserve their anchors;
- checked-in `.github/workflows/publish.yml` exactly equals generated output.

## Adding another platform family

Do not add package names or filename rules to the workflow generator.

A new family requires a generic contract:

1. define and export a canonical target-name glob in `@smoothbricks/nx-plugin`;
2. decide which runner can execute it;
3. exclude it from ordinary aggregate `build` only when it is platform-only;
4. include it in resolved dependency-closure validation;
5. teach managed publish generation to emit a producer only when resolved metadata contains a match;
6. collect only declared target outputs;
7. transfer outputs through a SHA-bound manifest;
8. add generator, no-match, dependency-closure, tamper, and checked-in parity tests.

If a platform needs a different architecture of the same OS, use an architecture-specific runner or a target-owned
cross-compilation command. Keep cache keys architecture-aware either way.

## Local verification

Run lint before tests:

```sh
nx lint nx-plugin
nx test nx-plugin

nx lint cli
nx test cli

smoo monorepo validate
```

For a package that owns native targets, also run the host-compatible targets directly and inspect the packed tarball
through the package's existing packed-package validation. Direct `bun test` is diagnostic-only; normal validation goes
through Nx so dependencies, preloads, tracing, and typechecking are wired correctly.
