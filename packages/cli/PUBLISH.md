# Three-step release process

This document describes the release workflow generated by `@smoothbricks/cli`. It is package-agnostic: repositories opt
in by declaring platform-family Nx targets and exact target outputs. The generator must not contain package names,
native filenames, or package-specific build commands.

The process has three release gates:

1. form and build the exact release candidate;
2. restore, assemble, and validate that candidate on Linux;
3. publish and optionally deploy the already-validated candidate.

For repositories with Apple-platform targets, gates 1 and 2 run as two GitHub Actions jobs: `macos-release-candidate`
followed by `publish-on-linux`. Gate 3 runs at the end of `publish-on-linux`, after every validation succeeds.
Repositories without Apple-platform targets keep the compact single-job Ubuntu workflow.

## Invariants

A release is valid only when all of these remain true:

- `smoo release version` runs exactly once for the candidate being published.
- Every producer builds the same release commit, not merely the workflow dispatch commit.
- Cross-job build outputs move through immutable workflow artifacts, never through Nx cache restoration.
- Every transferred file is declared by its resolved Nx target's `outputs` metadata.
- Every transferred file is bound to the candidate Git SHA and verified by size and SHA-256 before it enters the
  publishing worktree.
- Paths, symlinks, undeclared files, duplicate destinations, and destination collisions fail closed.
- The final Ubuntu job rebuilds or explicitly restores every file that enters the package tarball.
- Release validation does not honor ordinary-CI `ci:skip:<target>` tags.
- Only the final publishing job receives trusted-publishing/OIDC capability.
- The generated `# Step N` comments remain aligned with GitHub's job-local step numbering because external deeplinks use
  those anchors.

## Platform target contract

Smoo discovers platform work from resolved Nx project metadata. A target opts into a platform family through its exact
target name:

| Target-name glob | Producer                    | Purpose                                                            |
| ---------------- | --------------------------- | ------------------------------------------------------------------ |
| `*-macos`        | macOS release-candidate job | macOS-only output                                                  |
| `*-ios`          | macOS release-candidate job | Apple/iOS output requiring macOS                                   |
| `*-linux`        | final Ubuntu job            | supplemental Linux output excluded from ordinary aggregate `build` |

These families are intentionally excluded from the ordinary aggregate `build` dependencies. Otherwise a generic Linux
`nx build` could try to execute Apple targets, or an ordinary local build could unexpectedly build every release
platform.

Each platform target must:

1. have a real executor and command;
2. be cacheable when its command is deterministic;
3. declare precise workspace-relative `outputs` owned by that target;
4. name only target dependencies that are valid on the same runner family;
5. write platform-qualified filenames when multiple targets contribute to one package;
6. avoid a package-local no-op aggregate `build` target.

Example shape:

```json
{
  "nx": {
    "targets": {
      "native-arm64-macos": {
        "executor": "nx:run-commands",
        "cache": true,
        "inputs": ["{projectRoot}/src/**/*.rs", "{projectRoot}/Cargo.toml"],
        "outputs": ["{projectRoot}/dist/addon.darwin-arm64.node"],
        "options": {
          "command": "build-native --target aarch64-apple-darwin --output dist/addon.darwin-arm64.node"
        }
      },
      "native-arm64-linux": {
        "executor": "nx:run-commands",
        "cache": true,
        "inputs": ["{projectRoot}/src/**/*.rs", "{projectRoot}/Cargo.toml"],
        "outputs": ["{projectRoot}/dist/addon.linux-arm64-gnu.node"],
        "options": {
          "command": "build-native --target aarch64-unknown-linux-gnu --output dist/addon.linux-arm64-gnu.node"
        }
      }
    }
  }
}
```

The commands are illustrative. Smoo owns target discovery, execution, transfer, and validation; the package owns the
compiler invocation and its supported target triples.

## Gate 1: form the release candidate on macOS

When any selected release project owns a `*-macos` or `*-ios` target, the generated workflow starts with
`macos-release-candidate`.

The job:

1. checks out the workflow dispatch commit with full history;
2. restores the macOS/runner-architecture toolchain caches and enters devenv;
3. configures the release author and builds the self-hosted CLI when required;
4. repairs any previously interrupted release state;
5. runs `smoo release version` and captures its `mode`, selected projects, and resulting Git SHA;
6. checks generated/managed monorepo files against the versioned candidate;
7. resolves and runs matching `*-macos` and `*-ios` targets for only the selected release projects;
8. collects only those targets' declared output files;
9. creates a Git bundle containing the versioned candidate and release tags;
10. uploads the Git bundle and platform-output directory as short-lived workflow artifacts.

The platform invocation is metadata-driven:

```sh
smoo github-ci nx-run-many \
  --targets "*-macos,*-ios" \
  --projects "$RELEASE_PROJECTS" \
  --collect-outputs "$RUNNER_TEMP/macos-platform-outputs"
```

`nx-run-many` first expands the globs against `nx show project --json` metadata. It groups exact targets by their owning
projects instead of asking Nx to interpret wildcards. A clean no-match is a successful no-op; a target failure remains
fatal.

### Candidate identity

The producer records `git rev-parse HEAD` after versioning. That SHA is embedded in every output manifest and exposed as
a job output. The Git bundle—not the original checkout and not an inferred version—is the source of truth for the next
job.

Dry runs and `mode: none` still transfer the candidate Git state needed by the dependent job, while platform builds and
output upload remain conditional on a real release.

## Gate 2: restore, assemble, and validate on Ubuntu

`publish-on-linux` depends on `macos-release-candidate`, so it cannot observe a partially produced candidate.

The job:

1. checks out the dispatch commit only to obtain the repository and local actions;
2. downloads the candidate Git bundle and platform outputs;
3. fetches the bundle and resets the worktree to the recorded release commit before devenv setup;
4. enters the Linux/runner-architecture devenv and rebuilds the self-hosted CLI when required;
5. runs the ordinary aggregate `build`, `lint`, and `test` targets for the selected projects;
6. resolves and runs supplemental `*-linux` targets;
7. verifies and applies the macOS-produced outputs to their declared workspace destinations;
8. runs `smoo monorepo validate` against the fully assembled worktree.

The Linux family invocation is also metadata-driven:

```sh
smoo github-ci nx-run-many \
  --targets "*-linux" \
  --projects "$RELEASE_PROJECTS"
```

Ordinary `build` remains responsible for the final runner's normal host output. `*-linux` is only for additional Linux
artifacts that the aggregate must not build everywhere, such as another architecture.

### Artifact verification

`--collect-outputs` writes an exact Typia-validated manifest next to the staged files. Each file entry contains:

- owning Nx project;
- exact resolved target;
- declared output expression;
- workspace-relative destination path;
- candidate source SHA;
- byte size;
- SHA-256 digest.

The final job applies the artifact only with the expected release SHA:

```sh
smoo github-ci apply-outputs \
  --source-sha "$RELEASE_SHA" \
  "$RUNNER_TEMP/publish-artifacts/publish-macos-outputs-$GITHUB_RUN_ID"
```

Application rejects malformed or extra manifest properties, stale SHAs, checksum/size mismatches, missing or undeclared
files, duplicate destinations, existing destination collisions, absolute paths, traversal, and symlinks in either the
artifact tree or destination path. Verification completes before any file is copied into the publishing worktree.

### Caches are not artifact transport

Nix/devenv, `node_modules`, and Nx cache keys include both `runner.os` and `runner.arch`. This prevents one architecture
from restoring incompatible native state from another.

Those caches accelerate setup and repeated tasks; they do not move release outputs between jobs. GitHub workflow
artifacts carry the candidate bundle and declared platform outputs. The release remains correct when every cache misses.

## Gate 3: publish and deploy the validated candidate

Publishing is the final gate in `publish-on-linux`. It runs only after the assembled worktree passes monorepo
validation.

The job then:

1. packs public packages from the assembled release worktree;
2. validates packed manifests and package contents;
3. publishes with npm trusted publishing/OIDC, or reports the dry-run result;
4. pushes the prepared release commit and tags according to the release command's transaction rules;
5. optionally deploys production only after a non-dry-run release succeeds.

No producer job receives `id-token: write`. Native compilers and artifact upload steps therefore cannot publish packages
even if compromised.

A native package must include all required platform-qualified files in its packed tarball and select them by
`process.platform` and `process.arch` at runtime. Package-specific tests must prove every claimed supported platform
maps to exactly one included filename and that missing/unsupported combinations fail explicitly.

## Repositories without Apple targets

If no selected workspace target matches `*-macos` or `*-ios`, the generator retains the existing single Ubuntu `publish`
job. The same logical gates execute inline:

1. version the candidate;
2. run ordinary build plus matching `*-linux`, lint, tests, and monorepo validation;
3. publish and optionally deploy.

No empty producer job or artifact round-trip is generated. A repository with no platform-family targets retains its
previous workflow shape.

## Ordinary CI skip tags

A project may skip an exact target only in ordinary `nx-smart` CI:

```json
{
  "nx": {
    "tags": ["ci:skip:test"]
  }
}
```

Rules:

- the form is `ci:skip:<exact-target>`;
- the target must exist on that resolved project;
- the policy is generic—`build`, `lint`, `test`, or another exact target can be named;
- `smoo monorepo validate` rejects malformed tags and missing targets;
- explicit release `nx-run-many` calls do not honor the tag.

This permits a temporary ordinary-CI exclusion without silently removing the same package from release gates.

## Generated workflow step anchors

GitHub inserts “Set up job” as step 1. Generated workflow comments therefore begin checkout at `# Step 2`.
Composite-action internals do not increment the surrounding job's top-level step number.

The comments are a compatibility contract, not decoration. External logs and deeplinks use the job-local step number.
Any renderer change that inserts, removes, or conditionally emits a top-level step must regenerate every following
`# Step N` comment and update exact rendering tests. A skipped conditional step still exists in the job definition and
therefore still occupies its number.

Before accepting a generated workflow change, verify:

- every top-level step has the correct preceding `# Step N` comment;
- numbering restarts for each job;
- the automatic GitHub step remains documented as step 1;
- single-job and platform workflows both preserve their anchors;
- checked-in `.github/workflows/publish.yml` exactly equals generated output.

## Adding another platform family

Do not add package names or filename rules to the workflow generator.

A new family requires a generic contract:

1. define and export a canonical target-name glob in `@smoothbricks/nx-plugin`;
2. decide which runner can execute it;
3. exclude it from ordinary aggregate `build` only when it is platform-only;
4. include it in resolved dependency-closure validation;
5. teach managed publish generation to emit a producer only when resolved metadata contains a match;
6. collect only declared target outputs;
7. transfer outputs through a SHA-bound manifest;
8. add generator, no-match, dependency-closure, tamper, and checked-in parity tests.

If a platform needs a different architecture of the same OS, use an architecture-specific runner or a target-owned
cross-compilation command. Keep cache keys architecture-aware either way.

## Local verification

Run lint before tests:

```sh
nx lint nx-plugin
nx test nx-plugin

nx lint cli
nx test cli

smoo monorepo validate
```

For a package that owns native targets, also run the host-compatible targets directly and inspect the packed tarball
through the package's existing packed-package validation. Direct `bun test` is diagnostic-only; normal validation goes
through Nx so dependencies, preloads, tracing, and typechecking are wired correctly.
