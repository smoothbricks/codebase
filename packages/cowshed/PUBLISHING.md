# Publishing Cowshed

## Goal

Publish one `@smoothbricks/cowshed` npm package containing every native binary the package claims to support. The
package must never be published from a single host build: a local or ordinary Nx build intentionally produces only the
current platform's binary.

The initial supported target set is:

| Runtime           | Rust target                 | Package artifact                    |
| ----------------- | --------------------------- | ----------------------------------- |
| macOS arm64       | `aarch64-apple-darwin`      | `dist/cowshed.darwin-arm64.node`    |
| macOS x64         | `x86_64-apple-darwin`       | `dist/cowshed.darwin-x64.node`      |
| Linux arm64 glibc | `aarch64-unknown-linux-gnu` | `dist/cowshed.linux-arm64-gnu.node` |
| Linux x64 glibc   | `x86_64-unknown-linux-gnu`  | `dist/cowshed.linux-x64-gnu.node`   |

Windows is not in this matrix because the public Cowshed API transfers Unix file descriptors and the N-API crate uses
`std::os::fd`. Linux musl is not in the initial contract. Adding either platform requires runtime support, a loader
case, a native smoke test on that runtime, and a required release artifact; it must not be represented as supported
before those exist.

## Package and loader contract

`packages/cowshed/package.json` owns the NAPI-RS binary name and target list. `@napi-rs/cli` turns Cargo's
platform-specific `cdylib` into the correctly named `.node` file; no repository script renames Cargo outputs.

The normal local build remains:

```sh
nx run cowshed:cargo-napi
```

It invokes `napi build --platform` and creates exactly the current host artifact. `packages/cowshed/src/native.ts`
derives the required filename from `process.platform` and `process.arch`, then loads that file. `COWSHED_NODE_PATH`
remains the explicit development/test override. There is no universal `dist/cowshed.node` fallback because such a
filename hides which platform produced the package.

All four release artifacts deliberately live in the same `dist/` directory and are included by the existing package
`files: ["dist"]` rule. Do not adopt NAPI-RS's optional-dependency/per-platform-package template: this package's release
contract is one npm tarball containing the complete matrix.

## Nx target shape

Keep `cargo-napi` as the local host build and add four named Nx configurations before implementing the release workflow:

- `darwin-arm64`
- `darwin-x64`
- `linux-arm64-gnu`
- `linux-x64-gnu`

Each configuration must pass an explicit `--target <rust-target>` to `napi build`. This makes the requested target part
of the Nx task identity; relying on the runner's implicit host platform risks restoring the wrong artifact from cache.
Each matrix job starts with an empty artifact output directory and accepts exactly one `.node` output.

The `cargo-napi` inputs must continue to cover Rust sources, Cargo manifests and lock/config files, `package.json`, and
`bun.lock`. The target output remains `dist/**/*.node`. Toolchain selection files from `tooling/direnv/` must also be
added to the target inputs when cross-target configurations are introduced so a Rust/NAPI toolchain change invalidates
Nx cache entries.

## GitHub Actions design

Do not hand-edit only `.github/workflows/ci.yml`: it is generated by `packages/cli/src/monorepo/ci-workflow.ts`, and
monorepo validation will replace or reject drift. Add a separate repo-owned native release workflow and modify the
managed workflow generator only for the temporary test exclusion described below.

### Native build workflow

Add `.github/workflows/cowshed-native.yml` with `workflow_call` support and path-filtered pull-request/manual triggers
while the pipeline is being established. It has three stages.

### 1. Build matrix

Use native OS families rather than trying to build Apple binaries on Linux.

**macOS job**

- Runner: a pinned GitHub macOS image with Xcode and both Apple Rust targets installed.
- Matrix: `darwin-arm64`, `darwin-x64`.
- Run the matching `cowshed:cargo-napi:<configuration>` target.
- A macOS host may cross-compile between Apple arm64 and x64 using the installed Xcode SDK. Do not use Linux-to-macOS
  cross-compilation: Cowshed's dependency graph includes native code (notably `ring`), and a Linux build would require
  an externally supplied Apple SDK/toolchain with licensing and reproducibility concerns.
- Run the Node/Bun native smoke tests only when the artifact architecture matches the runner architecture. The other
  matrix entry is compile-and-inspect only.

**Linux job**

- Runners: native x64 and native arm64 GitHub runners when available, one target per runner.
- Matrix: `linux-x64-gnu`, `linux-arm64-gnu`.
- Run the matching `cowshed:cargo-napi:<configuration>` target.
- Run the Cowshed native smoke tests on both artifacts because each is built on a matching host.
- If a native arm64 runner is unavailable, use NAPI-RS `--use-napi-cross` on Linux and keep the glibc 2.17 floor. `ring`
  must compile with the selected cross compiler in CI before this fallback is accepted; a successful link alone does not
  replace the arm64 runtime smoke test.

Each matrix entry must:

1. Use `./.github/actions/setup-devenv` so Bun, Rust, Cargo, NAPI-RS dependencies, and repository PATH wiring match
   normal CI.
2. Build the TypeScript package and one explicitly configured native target.
3. Assert the exact expected filename exists and that no second `.node` file exists.
4. Inspect the binary architecture and dynamic dependencies (`file`, plus `otool -L` on macOS or `readelf`/`ldd` on
   Linux).
5. Run `nx test cowshed` only for an artifact executable on that runner.
6. Upload one artifact named `cowshed-native-<configuration>` containing only the `.node` file plus a checksum file.

A small composite action such as `.github/actions/build-cowshed-native/action.yml` may hold steps 2–6 so the macOS and
Linux jobs cannot drift. It must accept the Nx configuration and expected filename as explicit inputs; it must not infer
a release target from the host.

### 2. Assemble and verify

An Ubuntu assembly job downloads all four workflow artifacts into a clean `packages/cowshed/dist/` after the TypeScript
build. Before packing, it must fail on:

- a missing required filename;
- an unexpected or duplicate `.node` filename;
- a checksum mismatch;
- an architecture or binary-format mismatch;
- unresolved shared-library dependencies that are not part of the target OS contract.

The job then runs the existing packed-package validation and creates the npm tarball. Inspect the tarball contents and
assert that all four exact paths occur once. The assembly job never compiles a native binary and never treats a cached
local `dist/` directory as release input.

### 3. Publish

Wire the native workflow into the generated publish workflow before the `@smoothbricks/cowshed` publish step. The
publish step consumes the verified tarball artifact; it must not run `nx build cowshed` again, because that would
replace or contaminate the assembled matrix with the publisher host's single binary.

Only after the matrix and assembled-tarball checks pass:

1. remove `"private": true` from `packages/cowshed/package.json`;
2. add the repository's `npm:public` Nx tag;
3. add the intended `publishConfig` and release ownership metadata;
4. include Cowshed in the existing `smoo` release-project selection;
5. publish through the existing trusted-publisher/provenance path.

The release job must download by the producing workflow run ID and immutable artifact names. It must not select “latest
successful” native artifacts from another commit.

## Temporary CI unit-test exclusion

Nx can exclude the project now:

```sh
nx affected -t test --exclude=cowshed
```

The current managed CI invokes `smoo github-ci nx-smart`, which does not expose an exclude option. The clean temporary
implementation is:

1. add `--exclude <projects>` to `smoo github-ci nx-smart`;
2. pass it through to both its affected and run-many Nx command paths;
3. render `--exclude cowshed` for the Unit Tests step from `packages/cli/src/monorepo/ci-workflow.ts`;
4. regenerate `.github/workflows/ci.yml` with `smoo monorepo init`;
5. cover the passthrough and generated YAML in the CLI tests.

Do not use `describe.skip`, `CI` checks in Cowshed tests, or a successful no-op Cowshed test target. Those approaches
hide local behavior and can make Nx cache a false successful test result. The exclusion should be removed in the same
change that makes the Linux native build/smoke job required.

This exclusion skips the Cowshed `test` target only. The normal Linux CI build may continue compiling the host N-API
artifact, which catches Rust and link regressions while the release matrix is being completed.

## Cache changes

The existing Nx cache persists task outputs but does not persist Cargo's reusable compilation graph across GitHub-hosted
runs. Before enabling the four-target required workflow:

- configure `sccache` as Cargo's `RUSTC_WRAPPER` in devenv/CI;
- use an OS-, architecture-, Rust-toolchain-, and Cargo-lock-aware cache namespace;
- keep Apple and Linux caches separate;
- keep target directories or sccache entries separate by Rust target triple;
- never share a produced `.node` file through the compiler cache—only through the workflow artifact for that exact
  commit and target.

Nx configuration names plus OS-specific GitHub cache keys prevent cross-target artifact restoration. The assembly job
should not restore the Cowshed native Nx output cache at all.

## Required release gates

Publishing remains disabled until all of these are enforced by CI:

- four target builds succeed from a clean checkout;
- runnable target artifacts pass the existing Cowshed native test suite;
- every artifact's format, architecture, shared libraries, and checksum are verified;
- the assembled tarball contains all four binaries exactly once;
- the tarball's JavaScript and declarations pass packed-package validation;
- installing the tarball on macOS arm64, macOS x64, Linux arm64 glibc, and Linux x64 glibc loads the expected filename
  without `COWSHED_NODE_PATH`;
- deleting any one artifact makes assembly fail;
- the temporary main-CI Cowshed test exclusion has been removed.
