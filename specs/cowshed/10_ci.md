# cowshed as a CI runner substrate

cowshed turns a Linux+ZFS host into a self-hosted GitHub Actions runner where every repository-controlled workflow
command runs in an ephemeral, sandboxed, warm workspace — a ZFS clone of a headless main that already has Nix, the
package store, dependencies, and build caches hot. The workflow keeps the same job body it runs on stock GitHub images;
the runner integration selects and enforces the execution path, while a composite action provisions the matching
environment.

Repository binding is explicit and machine-independent. The runner configuration records a chosen remote URL and its
stable `repo_id`, normalized from that URL to lowercase `owner/repo`; startup validates that the two agree before a
runner registers or a warm main is opened. Discovery may propose a binding but never silently creates one. A repository
may have several identities, exactly one marked primary; a local-only repository must be configured with an explicit
`repo_id`.

## Runner host

A NixOS host provides, as one closure:

- **ZFS pool** with a `cowshed` container dataset (09_substrates.md).
- **Multi-user Nix daemon** and a warm `/nix/store` (the store the workspaces read; store reads need no grant —
  04_sandbox.md).
- **cowshed + cowshed-helper + cowshed-gateway**, gateway secrets from `systemd-creds` (registry tokens, a GitHub
  App/PAT for the git broker) — never from GitHub.
- **Ephemeral `actions-runner` instances** registered by the NixOS module `services.cowshed-runner`.

### `services.cowshed-runner`

```nix
services.cowshed-runner = {
  enable = true;
  repository = {
    remote = "https://github.com/smoothbricks/smoothbricks.git";
    repoId = "smoothbricks/smoothbricks";
    primary = true;
  };
  count = 4;                                  # concurrent ephemeral runners
  labels = [ "self-hosted" "cowshed" "zfs" ];
  project = "smoothbricks";                   # cowshed project whose main is cloned per job
};
```

The module stores trusted project policy at `~/.cowshed/<owner>/<repo>/policy.json`, with the `owner` and `repo`
components encoded independently for a path-safe layout. Here `~` is the cowshed controller identity's home, never the
ephemeral runner unit's home. That file is controller-owned, created only by trusted host bootstrap, and unavailable to
the runner unit and every job sandbox. Neither checkout content, workflow input, discovery, nor a job hook may create or
rewrite a binding or policy. A missing or inconsistent binding is a bootstrap error, not a cue to mint an identity from
the current machine path.

Each runner is `--ephemeral` (one job, then it deregisters and the unit restarts a fresh one). Per job, cowshed-runner's
job hooks and command interceptor do:

1. `ACTIONS_RUNNER_HOOK_JOB_STARTED`: `cowshed new ci-<run_id>-<attempt>` — a ZFS clone of `<project>/main` (tens of
   ms), sandbox baseline applied, env wiring emitted. The runner's work directory for the job is relocated to the
   workspace mount as a convenience, not as a confinement boundary.
2. Before the runner launches any repository-controlled workflow command — including the first — its command interceptor
   replaces the direct spawn with `cowshed exec` in that workspace. There is no direct-runner fallback.
3. `ACTIONS_RUNNER_HOOK_JOB_COMPLETED`: `cowshed rm ci-<run_id>-<attempt>` — destroys clone + origin snapshot regardless
   of job outcome. No state survives a job.

### Keeping the host main warm

The dev-machine convention "main is warm because you work in it" has no operator on a headless host, so its natural
stand-in is the main-branch CI job itself. After the required jobs pass on a push to the default branch, a
`cowshed-runner refresh` step (guarded exactly like the existing `Save Nx cache` step) fast-forwards `<project>/main` to
the merged SHA and re-runs install/build/test **inside the host main dataset**. The next job's clone inherits that
state. A refresh that fails to go green does not promote — the previous warm main stays authoritative, and
`cowshed doctor` on the host reports the lag (same mechanism as a dev machine).

## What this removes from the workflow

| Stock-image step                            | On a cowshed runner                                                                |
| ------------------------------------------- | ---------------------------------------------------------------------------------- |
| Install Nix, restore Nix store from cache   | gone — host daemon + warm store                                                    |
| `devenv`/direnv setup, restore devenv cache | gone — `.devenv` is in-dataset, arrives with the clone                             |
| `bun install`                               | gone — `node_modules` + bun cache in-dataset (03_caches.md)                        |
| Restore/Save Nx cache via `actions/cache`   | gone — `.nx` in-dataset, warm from main                                            |
| Registry/git secrets in GitHub secrets      | gone — gateway injects them host-side; job sees only the job-scoped `GITHUB_TOKEN` |

The job keeps checkout (a fast-forward inside the warm clone), actual build/lint/test targets, and artifact upload.
Uploads consume ordinary workspace artifacts or an explicit post-terminal `stdout_copy`/`stderr_copy` publication from a
sealed protected job artifact. The publication is an independent clone/reflink/copy, never a hardlink; CI never uploads
through `.cowshed/job/**`, a `Redirect.source`, or a path treated as job authority. Failure to publish is a separate
typed step failure and cannot rewrite the command's process/quota result.

## Composite action

`.github/actions/smoothbricks-ci/action.yml` (contract; the file lives in the consuming repo):

```yaml
inputs:
  mode: # auto | github | cowshed
    default: auto
outputs:
  workspace-path: # cwd for subsequent steps; the repo root in both modes
```

- **`mode: auto`** resolves to `cowshed` when the runner advertises the `cowshed` label and the `cowshed` binary is on
  `PATH`, else `github`.
- **`github`** wraps the existing `./.github/actions/setup-devenv` path — fast Nix install on GitHub's stock image,
  unchanged behavior. This is the public-PR and fallback path.
- **`cowshed`** assumes the job hook already created the workspace; it resolves the mount, runs `cowshed ensure --envrc`
  to export the env, and sets `workspace-path`.

The composite action provisions the environment and publishes a convenient cwd only. A completed action cannot rewrite
how the GitHub runner launches later steps, and cwd relocation does not apply Seatbelt/Landlock. On a cowshed runner,
wrapping later commands is exclusively the runner integration's responsibility.

The job body (`smoo github-ci nx-smart …` steps in `ci.yml`) is **byte-identical** across modes — mode only changes how
the environment is provisioned, never what runs.

### Command interception contract

For a cowshed-mode job, the runner integration MUST classify every workflow step before any repository-controlled
payload starts. Every supported step that can launch a repository-controlled command is dispatched through
`cowshed exec` with the job workspace; the runner MUST NOT spawn that command directly. This applies to the first step
and every later step, whether authored in the repository or injected by a fork PR.

An action type is supported only when the integration can intercept every repository-controlled process it may launch. A
step using an action type that cannot be completely intercepted MUST be rejected before its payload starts, with a
classifiable configuration error; it MUST NOT run directly, silently weaken confinement, or rely on cwd relocation.

## Public-repo security posture

Self-hosted runners on public repositories are dangerous by default: a fork PR can run arbitrary code on the runner.
cowshed's stack makes fork-PR execution **defensible, not risk-free**, and both halves matter:

**Where the sandbox boundary actually is.** Repository-controlled workflow commands are confined by cowshed because the
runner integration launches each supported command through `cowshed exec`; unsupported action types are rejected before
their payload starts. The composite action does not create this boundary. The **runner's own systemd unit remains
confined as defense in depth** against runner/interceptor defects and host-side runner behavior, and is therefore a hard
requirement of `services.cowshed-runner`, not optional hardening:

- the runner runs as a **dedicated unprivileged user** with no read/write access to controller-owned cowshed state
  (grant files, gateway socket/credentials, repository bindings, or `~/.cowshed/<owner>/<repo>/policy.json`) — enforced
  by unit sandboxing (`DynamicUser`/`ProtectHome`/`InaccessiblePaths` over those controller-owned paths); the narrow
  runner-to-controller interface permits workspace lifecycle and `cowshed exec` dispatch without exposing their backing
  files;
- the unit's **network is forced through the gateway** at the unit level (its own netns / egress restriction), so a
  runner or interceptor defect cannot restore arbitrary host egress;
- only `cowshed-helper` is root, and it refuses any dataset outside `<pool>/cowshed/`.

With the unit confined, the cowshed-side mitigations hold for all supported repository-controlled commands, while the
unit remains a second boundary around the runner itself:

- Ephemeral workspace **destroyed after every job** — no persistence across jobs, no cache poisoning that outlives a
  run.
- Closed-baseline sandbox on the job body — it writes only its own clone + designated caches; it cannot read the host,
  other jobs, cowshed state, or gateway credentials.
- **Egress only through the gateway allowlist** — enforced for repository-controlled commands by their `cowshed exec`
  profile and again around the runner by unit networking; neither cwd relocation nor the composite action is credited
  with this property.
- **No secrets reachable on disk or in env** — gateway holds registry/git credentials host-side; the job's only secret
  is the job-scoped `GITHUB_TOKEN`, which GitHub already restricts to read-only for fork PRs.
- Job diagnostics reconcile compact controller lifecycle/order/lineage commitments with protected in-volume content
  authority. The runner never copies raw stdout/stderr into controller telemetry, trusts mutable redirect/publication
  paths as evidence, or destroys a workspace after an integrity mismatch without preserving it for diagnosis.

Required GitHub-side settings (cowshed cannot enforce these — the operator must):

- **Require approval for all outside-collaborator fork-PR workflow runs.**
- **Ephemeral runners only**, in a **dedicated runner group scoped to the single repo** (never an org-wide group
  reachable by other repos' workflows).
- Concurrency limited to the runner count; no `pull_request_target` with a checkout of untrusted code.

Stated plainly: wrapping every supported repository-controlled command with `cowshed exec`, rejecting action types that
cannot be intercepted, and retaining runner-unit confinement reduces a fork-PR compromise from "host takeover + secret
theft" to "burn one ephemeral, sandboxed, secret-less clone." That is defensible for an open-source project that wants
warm CI without handing forks the keys — it is not a substitute for review of what fork PRs do, and neither cwd
relocation nor the composite action alone provides the sandbox boundary.

## GitHub-hosted images: the honest constraint

You cannot ship a NixOS image to GitHub-hosted runners on free/open-source plans — those runners boot GitHub's stock
Ubuntu/macOS/Windows images only, and the larger-runner **custom-image** preview that once allowed uploaded VM images
has been discontinued. There is no path to "GitHub-hosted NixOS." This is exactly why the action has two modes: `github`
is the best possible experience on a stock image (a fast Nix install, warm from `actions/cache`), and `cowshed` is the
self-hosted path that gets the warm-clone + sandbox + secrets-off-GitHub properties. A repo can run `github` on public
fork PRs (GitHub-hosted, fully sandboxed by GitHub) and `cowshed` on trusted branches, selected by `runs-on` labels and
the `auto` mode, with no change to the job body.

## Tradeoffs

**Self-hosted at all, vs staying fully GitHub-hosted.** GitHub-hosted runners need no operator and are perfectly
isolated per job, but reinstall Nix and re-restore every cache on every run and require every secret to live in GitHub.
cowshed runners move that cost to a warm host and move secrets off GitHub entirely, at the price of operating a host and
accepting the shared-kernel confinement ceiling (04_sandbox.md). The `auto` mode means adopting cowshed runners is never
all-or-nothing.

**Job-hook workspace creation vs an action step.** Creating the workspace in the runner's job hook (not a workflow step)
means the command interceptor can route the very first workflow command through `cowshed exec`, so a malicious first
step cannot pre-empt isolation. The cost — workspace lifecycle and command interception live in the runner host config,
not the visible workflow — is why `cowshed doctor` on the host surfaces orphaned `ci-*` workspaces from crashed runners
for the operator.
