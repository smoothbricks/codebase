A tracer turns op executions into emitted spans. Understanding the order of its lifecycle hooks explains where each
built-in tracer does its work ‚Äî and where a custom one plugs in.

## The flow of one trace

```text
trace(name, op)
   ‚îÇ  createTraceRoot()  ‚Üí timestamps + trace_id
   ‚ñº
allocate root buffer
   ‚îÇ  onTraceStart(rootBuffer)
   ‚ñº
run the op  ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∫  ctx.span(child)
   ‚îÇ                              ‚îÇ allocate child buffer
   ‚îÇ                              ‚îÇ onSpanStart(childBuffer)
   ‚îÇ                              ‚îÇ run child ‚Ä¶
   ‚îÇ                              ‚îÇ onSpanEnd(childBuffer)
   ‚ñº
op returns ok/err
   ‚îÇ  onTraceEnd(rootBuffer)   ‚Üê convert / collect / ship here
   ‚ñº
buffer strategy: toArrowTable / releaseBuffer
```

## The hooks

- **`onTraceStart` / `onTraceEnd`** fire at the boundaries of a root trace. `onTraceEnd` is where most tracers act ‚Äî
  `StdioTracer` prints, `SQLiteTracer` writes, `ArrayQueueTracer` enqueues.
- **`onSpanStart` / `onSpanEnd`** fire for each child span. Streaming tracers (like `StdioTracer`) use them to print as
  execution happens; collecting tracers leave them as no-ops.
- **`onStatsWillResetFor`** fires before a buffer's stats reset, so a tracer can capture them first.

## The buffer strategy

The hooks receive **buffers**, not Arrow tables. Converting and freeing them is the buffer strategy's job
(`toArrowTable`, `releaseBuffer`), which the tracer owns as `this.bufferStrategy`. This separation is why the same
tracer logic works regardless of how buffers are allocated.

To implement your own, you fill in these hooks ‚Äî see [Write a custom tracer](/guides/write-a-custom-tracer/). For the
big picture, see [Architecture overview](/concepts/architecture/).
