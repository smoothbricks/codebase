The idiomatic pattern for a web framework (Express, Fastify, Hono, …) is to open **one trace per request** at the
handler boundary and use child spans for the steps inside.

The part worth getting right is **who records what**:

| Concern | Owner | Why |
|---|---|---|
| `requestId`, route, HTTP status | the **handler** | These are facts of the HTTP boundary. The framework is what knows them. |
| Domain outcome (`ok` / `err`), domain attributes | the **op** | An op should be callable from a queue worker or a cron job, where "HTTP 404" is meaningless. |

## The domain op knows nothing about HTTP

It reports *what happened*, not *how to respond*:

```ts
const loadCart = defineOp('load-cart', async (ctx, userId: string) => {
  const cart = await db.carts.find(userId);
  if (!cart) return ctx.err(CART_NOT_FOUND({ userId }));

  return ctx.ok(cart).with({ itemCount: cart.items.length });
});
```

No `status`, no route, no request id — so the same op works unchanged behind a queue consumer.

## The handler owns the request-level facts

Open the trace at the boundary. Pass request-scoped values through `trace`'s overrides, `setScope` them so every child
span inherits them, and map the domain outcome onto HTTP **at the point the outcome is known**:

```ts
// Express-style handler
app.get('/cart', async (req, res) => {
  const result = await tracer.trace(
    'GET /cart',
    { requestId: req.headers['x-request-id'] }, // userCtx overrides
    async (ctx) => {
      ctx.setScope({ requestId: ctx.requestId }); // inherited by every child span
      ctx.tag.route('/cart'); // the handler is what knows the route

      const cart = await ctx.span('load-cart', loadCart, req.userId);

      return cart.success
        ? ctx.ok(cart.value).status(200)
        : ctx.err(cart.error).status(404);
    },
  );

  res.status(result.success ? 200 : 404).json(result.success ? result.value : { error: true });
});
```

Here `requestId` is a user-context property declared in
`defineOpContext({ ctx: { requestId: undefined as string | undefined } })`, so it's available as `ctx.requestId`.

:::caution[Don't tag the status up front]
`ctx.tag.status(200)` at the top of an op records a 200 even when the request goes on to fail — the tag is written
before the outcome exists. Put the status on the **result** instead, where it can differ per branch.

Schema fields are setters on the result too, mirroring `ctx.tag` on row 0: `ctx.ok(v).status(200)` writes directly to
the span-end row. Setting several fields at once still goes through `.with({ … })` — `ctx.ok(v).with({ status: 200, requestId })`
— the single-field form above is just the one-field case of the same mechanism.
:::

## What the trace looks like

The split shows up in the emitted rows. The child span records the domain failure with no status; the boundary records
the HTTP mapping:

```text
span-start  route=/cart   status=null
span-start  (load-cart)   status=null
span-err    (load-cart)   status=null   ← domain op: what happened
span-err    route=/cart   status=404    ← handler: how we responded
```

## Why scope, not repeated tags

Setting `requestId` once via `setScope` means each child span carries it automatically — you avoid threading the value
through every function. See [tag vs scope vs log](/guides/tag-vs-scope-vs-log/).

:::note
To emit the resulting traces somewhere, swap the tracer: `SQLiteTracer` to persist, `ArrayQueueTracer` to collect
buffers for Arrow conversion, or `CompositeTracer` to do several at once. See
[Persist traces to SQLite](/guides/persist-to-sqlite/), [Batch and send traces](/guides/batch-and-send-traces/), and
[Convert a trace to Arrow](/guides/convert-to-arrow/).
:::
