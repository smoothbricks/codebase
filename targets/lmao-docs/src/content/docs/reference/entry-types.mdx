Each row in a span buffer carries an **entry type** that says what the row represents. You usually don't touch these
directly — `tag`, `log`, `ok`/`err`, and feature-flag access write the right rows for you — but they're useful when
reading raw trace output.

## Entry-type names

| Group | Names |
|---|---|
| Span lifecycle | `span-start`, `span-ok`, `span-err`, `span-exception`, `span-retry` |
| Log levels | `trace`, `debug`, `info`, `warn`, `error` |
| Feature flags | `ff-access`, `ff-usage` |
| Period marker | `period-start` |
| Op metrics | `op-invocations`, `op-errors`, `op-exceptions`, `op-duration-total`, `op-duration-ok`, `op-duration-err`, `op-duration-min`, `op-duration-max` |
| Buffer metrics | `buffer-writes`, `buffer-spans`, `buffer-capacity` |

The matching numeric constants are exported as `ENTRY_TYPE_*` (e.g. `ENTRY_TYPE_SPAN_OK`, `ENTRY_TYPE_INFO`). The
lifecycle and log values are:

| Constant | Value | Constant | Value |
|---|---|---|---|
| `ENTRY_TYPE_SPAN_START` | 1 | `ENTRY_TYPE_TRACE` | 6 |
| `ENTRY_TYPE_SPAN_OK` | 2 | `ENTRY_TYPE_DEBUG` | 7 |
| `ENTRY_TYPE_SPAN_ERR` | 3 | `ENTRY_TYPE_INFO` | 8 |
| `ENTRY_TYPE_SPAN_EXCEPTION` | 4 | `ENTRY_TYPE_WARN` | 9 |
| `ENTRY_TYPE_SPAN_RETRY` | 5 | `ENTRY_TYPE_ERROR` | 10 |

Feature-flag, period, op-metric, and buffer-metric constants continue from 11 upward. Compare against these constants
when filtering rows (see [Measure op performance](/guides/measure-op-performance/)).

## Reserved system columns

These columns are managed by LMAO and present in the output; your schema field names must not collide with them:

`trace_id`, `thread_id`, `span_id`, `parent_thread_id`, `parent_span_id`, `timestamp`, `entry_type`.

## Multi-purpose columns

A few columns carry different meanings depending on the entry type:

- **`message`** — a span name (lifecycle rows), a log message template (log rows), or a flag name (feature-flag rows).
- **`ff_value`** — a feature flag's resolved value.
- **`uint64_value`** — large 64-bit integers, used for metrics and big counts. Write your own from a log entry with
  `.uint64_value(value)` (the argument is a `bigint`) — see [SpanContext](/reference/span-context/#ctxlog).

See [Span identity](/concepts/span-identity/) for how the identity columns reconstruct the trace tree.
